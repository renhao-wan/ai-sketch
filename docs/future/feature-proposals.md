## 一、小型功能（1-3 天）

### 1.1 图表版本历史

**描述**：为每个对话记录图表代码的每次变更版本，支持回退到任意历史版本。

**实现思路**：
- 在 `messages` 表中，assistant 消息已保存了每次生成的代码
- 只需在 UI 层添加版本切换功能：点击历史消息中的"查看此版本"按钮，将画布切换到该版本的代码
- 在 `BottomContextPanel` 中添加版本时间线视图

**价值**：用户经常需要对比不同版本的图表，或回退到之前的效果。当前只能通过翻聊天记录实现，体验差。

**工作量**：1-2 天

---

### 1.2 图表导出为图片/PDF

**描述**：支持将画布上的图表导出为 PNG、SVG、PDF 格式。

**实现思路**：
- Excalidraw：已有 `exportToBlob` 和 `exportToSvg` API
- Mermaid：可通过 `mermaid.render` 生成 SVG
- Draw.io：iframe 内的 `postMessage` 协议支持导出
- 在 `FloatingToolbar` 或 `FloatingAIActions` 旁添加导出按钮

**价值**：当前仅支持导出为原始格式文件（.json/.md/.xml），用户无法直接在文档或演示中使用图表。

**工作量**：2-3 天

---

### 1.3 快捷键增强

**描述**：添加更多编辑器快捷键。

**建议的快捷键**：
| 快捷键 | 功能 |
|--------|------|
| `Ctrl+Enter` | 发送消息（当前仅 Enter） |
| `Ctrl+Shift+Enter` | 重新生成 |
| `Ctrl+1/2/3` | 切换格式（Excalidraw/Mermaid/Draw.io） |

| `Ctrl+L` | 清空当前对话 |
| `Ctrl+N` | 新建对话 |
| `Ctrl+Z` | 撤销代码编辑（Monaco 原生支持） |

**价值**：桌面端用户重度依赖快捷键，当前快捷键数量偏少。

**工作量**：1 天

### 1.7 消息气泡代码预览改进

**描述**：将 300 字符硬截断改为可折叠的代码块。

**实现思路**：
```tsx
const [expanded, setExpanded] = useState(false);
const isLong = message.content.length > 300;
const displayContent = expanded ? message.content : message.content.substring(0, 300);

<pre>
  <code>{displayContent}{isLong && !expanded && '...'}</code>
  {isLong && (
    <button onClick={() => setExpanded(!expanded)}>
      {expanded ? '收起' : `展开全部 (${message.content.length} 字符)`}
    </button>
  )}
</pre>
```

**价值**：当前用户完全无法在消息气泡中预览 AI 生成的代码。

**工作量**：0.5 天

### 1.8 面板尺寸持久化

**描述**：将左侧面板宽度和底部面板高度保存到 localStorage。

**实现思路**：
```typescript
const [panelWidth, setPanelWidth] = useState(() => {
  const saved = localStorage.getItem('editor-panel-width');
  return saved ? parseInt(saved) : 360;
});

useEffect(() => {
  localStorage.setItem('editor-panel-width', panelWidth.toString());
}, [panelWidth]);
```

**价值**：用户每次刷新页面都需要重新调整面板宽度。

**工作量**：0.5 天

## 二、中型模块（1-2 周）

### 2.1 图表模板市场

**描述**：内置丰富的图表模板库，按行业和场景分类，用户可一键使用。

**实现思路**：
- 在 `lib/templates/` 目录下按分类存储模板 JSON
- 每个模板包含：名称、描述、预览图（base64）、图表类型、格式、代码
- 首页添加"模板市场"入口，支持分类浏览和搜索
- 点击模板直接加载到编辑器

**建议的模板分类**：
| 分类 | 示例模板 |
|------|----------|
| 软件架构 | 微服务架构、分层架构、事件驱动、CQRS |
| 业务流程 | 订单流程、审批流程、用户注册、支付流程 |
| 数据库设计 | 电商 ER、社交网络 ER、CMS ER |
| 组织架构 | 公司组织图、项目团队、敏捷角色 |
| 思维导图 | 产品规划、技术选型、SWOT 分析 |
| 网络拓扑 | 云架构、VPN 拓扑、CDN 架构 |
| UI 流程 | 页面导航、用户旅程、表单流程 |

**价值**：当前仅 5 个固定模板，覆盖面窄。丰富的模板库能大幅降低新用户的使用门槛。

**工作量**：1 周（含模板制作）

### 2.2 代码编辑器双向同步

**描述**：代码编辑器中的修改能实时反映到画布，画布上的操作也能更新代码。

**实现思路**：
- **代码→画布**：Monaco Editor 的 `onChange` 事件，debounce 500ms 后调用 `tryParseAndApply`
- **画布→代码**：Excalidraw 的 `onChange` 回调，将 elements 序列化回 JSON 更新 `generatedCode`
- 需要处理循环更新问题：设置 `updateSource` 标记区分来源

**价值**：当前代码编辑器是单向的（代码→画布），用户在画布上的修改无法同步回代码，导致重新生成时丢失手动调整。

**工作量**：1-2 周

---

### 2.3 图表对比视图

**描述**：支持左右对比两个版本的图表，或对比不同格式的渲染效果。

**实现思路**：
- 在编辑器中添加"对比模式"按钮
- 左侧显示版本 A，右侧显示版本 B
- 版本可以是同一对话的不同消息，也可以是不同格式的渲染结果
- 使用 `SplitPane` 组件实现左右分栏

**价值**：用户在调整图表时需要对比前后效果，当前只能靠记忆。

**工作量**：1 周

**价值**：当用户需要为文档生成多张图表时，逐个生成效率低。

**描述**：允许用户创建自定义的 AI 操作（类似自定义 prompt）。

- "调整配色" — 将配色方案改为指定的色板

**价值**：当前 4 个 AI 操作是固定的，用户无法根据自己的需求定制。


**价值**：当前修改图表必须通过自然语言描述全局修改，无法精确操作单个元素。

## 三、大型模块（2-4 周+）

### 3.1 Agent 多步生成

**描述**：引入 Agent 工作流，支持多步推理和工具调用来生成复杂图表。

**核心思路**：

传统模式：用户描述 → LLM 一次性生成完整代码 → 渲染

Agent 模式：用户描述 → Agent 规划 → 分步生成 → 中间检查 → 迭代优化 → 最终渲染

**实现架构**：

```
用户输入
    ↓
Agent Planner（规划阶段）
  - 分析需求复杂度
  - 决定是否需要分步
  - 拆解为子任务列表
    ↓
Step Executor（执行阶段）
  - 按子任务逐个调用 LLM
  - 每步生成部分元素
  - 中间结果合并到画布
    ↓
Quality Checker（检查阶段）
  - 检查元素数量是否合理
  - 检查箭头连接是否正确
  - 检查布局是否美观
  - 如有问题，触发修复步骤
    ↓
Final Optimizer（优化阶段）
  - 整体布局优化
  - 样式统一
  - 导出最终结果
```

**支持的 Agent 模式**：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **骨架填充** | 先生成节点骨架 → 再填充细节 → 最后添加连接 | 大型架构图（20+ 节点） |
| **分区域生成** | 按功能区域分批生成 → 合并 → 统一布局 | 微服务架构、系统设计 |
| **迭代精化** | 生成初稿 → 识别问题 → 修复 → 重复直到满意 | 复杂流程图 |
| **参考图生成** | 分析参考图片 → 提取结构 → 生成类似图表 | 从截图重建图表 |

**关键技术点**：
1. **Agent Loop**：实现 `plan → execute → check → fix` 的循环
2. **工具注册**：给 Agent 提供工具（如 `addElements`、`moveElement`、`changeColor`）
3. **中间状态管理**：维护当前画布状态，每步增量更新
4. **成本控制**：限制最大步数和 token 消耗
5. **进度反馈**：SSE 流式返回每步的执行状态

**价值**：这是项目最大的差异化竞争力。当前市场上的 AI 图表工具都是一次性生成，Agent 模式能处理复杂得多的场景。

**工作量**：3-4 周

### 3.2 代码仓库图表生成

**描述**：分析代码仓库的结构，自动生成架构图、依赖关系图、ER 图等。

**实现思路**：
1. 用户输入 GitHub 仓库 URL 或本地路径
2. 后端克隆/读取代码，使用 AST 解析器分析：
   - 目录结构 → 架构图
   - import/require 依赖 → 依赖关系图
   - 数据库模型定义 → ER 图
   - API 路由定义 → API 流程图
   - 类定义 → 类图
3. 将分析结果作为上下文发送给 LLM
4. LLM 生成对应的图表代码

**支持的分析类型**：
| 输入 | 输出图表 |
|------|----------|
| 目录结构 | 分层架构图 |
| package.json / requirements.txt | 依赖关系图 |
| Prisma/SQLAlchemy 模型 | ER 图 |
| Express/FastAPI 路由 | API 流程图 |
| TypeScript 接口定义 | 类图 |
| Docker Compose | 部署架构图 |

**价值**：开发者经常需要为现有项目绘制架构图，手动绘制耗时且容易过时。

**工作量**：2-3 周

### 3.3 实时协作

**描述**：支持多人同时编辑同一张图表。

**实现思路**：
- 引入 WebSocket 服务器（或使用 WebRTC P2P）
- 使用 CRDT（如 Yjs）实现冲突解决
- Excalidraw 原生支持协作（Excalidraw+ 的核心功能）
- 每个用户有独立的光标颜色和名称标识

**功能清单**：
- 实时光标显示
- 多人同时编辑
- 聊天/评论功能
- 协作者列表
- 权限控制（查看/编辑/管理）

**价值**：团队协作场景下，多人需要共同评审和修改图表。

**工作量**：3-4 周（不含基础设施）

### 3.4 图表知识库

**描述**：将用户的图表、模板、代码片段组织为可检索的知识库。

**实现思路**：
- 新建 `knowledge_base` 表，存储结构化的图表元数据
- 对图表内容进行向量化嵌入（使用本地 embedding 模型或 API）
- 支持语义搜索："找一下上次画的那个微服务架构图"
- 支持标签分类和文件夹组织
- 支持导出为 Markdown 文档

**价值**：随着图表数量增长，用户需要高效地检索和复用历史图表。

**工作量**：2-3 周

### 3.5 图表自动更新

**描述**：当源数据变化时，自动重新生成图表。

**应用场景**：
- 监控架构图：从 Prometheus/Grafana 的指标自动更新服务健康状态
- 数据库 ER 图：当数据库 schema 变更时自动更新
- API 流程图：当路由文件变更时自动更新

**实现思路**：
- 支持配置"数据源"（GitHub webhook、数据库连接、API 端点）
- 数据源变化时触发重新生成
- 使用 diff 算法对比新旧版本，仅更新变化部分

**价值**：技术文档中的图表经常因为代码变更而过时。

**工作量**：3-4 周

---

### 3.6 多图表联动

**描述**：支持在同一个画布上放置多张相关图表，并建立联动关系。

**实现思路**：
- 画布支持"分区"模式，每个分区独立渲染一张图表
- 分区之间可以用连线表示关系（如"详细展开"、"数据流向"）
- 支持从总览图点击某个节点，跳转到对应的详细图
- 类似"无限画布 + 子页面"的概念

**价值**：复杂系统的设计通常需要多张图表互相补充，当前每张图表是独立的。

**工作量**：2-3 周

---

### 3.7 图表动画

**描述**：支持为图表添加动画效果，用于演示和教学。

**动画类型**：
| 类型 | 说明 |
|------|------|
| **逐步展示** | 按顺序逐个显示节点和连线 |
| **高亮路径** | 动态高亮某条数据流或执行路径 |
| **缩放聚焦** | 从全景逐步聚焦到某个区域 |
| **状态变化** | 节点颜色/形状随时间变化 |

**实现思路**：
- 在 Excalidraw 之上叠加动画控制层
- 定义动画关键帧（时间、元素、属性变化）
- 使用 `requestAnimationFrame` 驱动动画
- 支持导出为 GIF/MP4

**价值**：技术分享和教学场景中，动画比静态图更直观。

**工作量**：2-3 周

### 3.8 本地模型支持

**描述**：集成 Ollama 等本地模型运行时，支持完全离线使用。

**实现思路**：
- 在设置页面添加"本地模型"配置选项
- 检测本地 Ollama 服务是否运行（默认 `http://localhost:11434`）
- 自动获取本地可用模型列表
- 调整 prompt 以适应小模型的能力（简化 system prompt、减少输出要求）

**推荐的本地模型**：
| 模型 | 适用场景 |
|------|----------|
| CodeLlama 7B | 代码生成，资源占用小 |
| Qwen2.5-Coder 7B | 中文友好，代码能力强 |
| DeepSeek-Coder 6.7B | 代码生成专用 |
| Llama 3.1 8B | 通用能力，需要更详细的 prompt |

**价值**：隐私敏感用户和离线场景需要完全本地化的解决方案。当前已支持 OpenAI 兼容 API（Ollama 兼容），但缺少针对本地模型的优化。

**工作量**：1-2 周（主要是 prompt 适配和 UI）

```
        │                    │                    │
        │           ┌────────▼────────┐           │
        │           │  快速编辑        │           │
        │           └────────┬────────┘           │
        │                    │                    │
┌───────▼────────────────────▼────────────────────▼───────┐

└───────┬────────────────────┬────────────────────┬───────┘
        │                    │                    │
┌───────▼───────┐   ┌────────▼────────┐   ┌───────▼───────┐

└───────────────┘   └─────────────────┘   └───────────────┘
```

**关键依赖**：
1. Agent 多步生成 依赖于 代码编辑器双向同步（需要中间状态管理）
2. 代码仓库分析 依赖于 Agent 多步生成（复杂分析需要多步推理）
3. 图表对比视图 依赖于 版本历史（需要多个版本的数据）
4. 自定义 AI 操作 依赖于 图表模板市场（模板可作为操作的参考）
