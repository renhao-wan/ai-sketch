# AI 响应缓存

## 简介

AI Sketch 内置了 L1/L2 分层智能响应缓存机制，能够记住之前生成过的图表代码。当您输入相同的描述时，系统会直接返回缓存的结果，无需等待 AI 重新生成。

缓存采用两级架构：

- **L1 内存缓存** — 速度极快，适合热点数据的即时访问
- **L2 SQLite 持久缓存** — 容量更大，跨会话持久化存储

两级缓存协同工作，兼顾速度与容量。

## 架构

### L1：内存缓存（MemoryCache）

- 基于内存的 LRU（Least Recently Used，最近最少使用）缓存
- 最大容量 **50 条目**，总体积上限 **1MB**
- 访问速度极快，命中时几乎零延迟
- 进程重启后清空

### L2：SQLite 持久缓存

- 存储在本地 SQLite 数据库的 `response_cache` 表中
- 最大容量 **100MB**
- 默认 TTL（Time To Live，生存时间）**7 天**，可在设置中配置
- 跨会话持久化，重启后仍有效
- 采用 LRU 淘汰策略：当缓存体积达到高水位（90%）时触发清理，清理至低水位（80%）

### Inflight 请求去重

系统会对正在进行中的相同请求进行去重。如果多个相同的请求同时发起，只有第一个请求会实际调用 AI，其余请求会等待并共享同一个结果，避免重复消耗 API 资源。

## 缓存键

系统使用以下信息组合生成统一的缓存标识：

- **prompt** — 用户输入的描述文本
- **format** — 图表格式（Excalidraw / Mermaid / Draw.io）
- **chartType** — 图表类型（流程图、架构图、思维导图等）
- **model** — 使用的 AI 模型
- **configName** — LLM 配置名称
- **contextHash** — 对话上下文的哈希值

只有当以上所有条件都完全匹配时，才会返回缓存结果。这确保了不同模型和配置下的生成结果不会混淆。

## 缓存条件

缓存功能在以下条件下自动生效：

1. **纯文本输入** — 上传图片的请求不会被缓存
2. **首次生成** — 重新生成（点击"重新生成"按钮）不会使用缓存
3. **多轮对话支持** — 多轮对话中的消息同样可以命中缓存（根据上下文哈希匹配）

## 缓存策略

| 参数 | 值 |
|------|-----|
| L1 最大条目 | 50 |
| L1 最大体积 | 1MB |
| L2 最大体积 | 100MB |
| 默认 TTL | 7 天（可配置） |
| 淘汰策略 | LRU（高水位 90% 触发，低水位 80% 目标） |

## 缓存失效

### 自动失效

以下情况会自动使相关缓存失效：

- **LLM 配置被删除** — 该配置下所有缓存自动清除
- **模型发生变更** — 更换模型后旧缓存不再匹配（因缓存键包含模型信息）

### 手动失效

在 **设置 → 缓存管理** 面板中，您可以：

- 查看缓存统计信息（条目数量、占用空间）
- 清除所有缓存
- 按指定配置清除缓存
- 调整 TTL 设置

## 用户体验

### 缓存命中时

当缓存命中时，您会看到：

1. 图表代码以流式方式快速输出（每块 100 字符，间隔 10ms），模拟真实生成体验
2. 控制台会显示 `[Generate] Cache hit` 日志

### 缓存未命中时

正常调用 AI 生成，生成结果会自动保存到缓存，供下次使用。

## 管理界面

在应用的 **设置 → 缓存管理** 中，您可以：

- **查看统计** — 当前缓存条目数、占用空间、命中率等
- **清除全部缓存** — 一键清除所有 L1 和 L2 缓存
- **按配置清除** — 仅清除特定 LLM 配置下的缓存
- **TTL 设置** — 调整缓存的有效期（默认 7 天）

## 常见问题

### Q: 为什么我的请求没有使用缓存？

可能的原因：

- 上传了图片（图片请求不缓存）
- 点击了"重新生成"按钮
- 更换了 AI 模型或 LLM 配置
- 修改了图表类型或格式
- 缓存已过期（超过 TTL 设置的时间）
- 缓存已被淘汰（LRU 策略清理）

### Q: 缓存会不会占用太多空间？

不会。L2 缓存设有 100MB 的体积上限，采用 LRU 淘汰策略自动管理。每个条目只保存文本代码（不含图片），占用空间很小。您也可以在设置中手动清除缓存。

### Q: 如何清除缓存？

在 **设置 → 缓存管理** 面板中，您可以：

- 点击"清除全部缓存"一键清理
- 选择特定配置进行按需清除

如果需要完全重置，也可以删除数据库文件（`data/ai-sketch.db`）。

### Q: 多轮对话也能使用缓存了吗？

是的。新版本支持多轮对话的缓存匹配，系统会根据完整的对话上下文哈希进行匹配。只要对话上下文一致，多轮对话中的重复请求同样可以命中缓存。

## 技术细节

### L2 缓存表结构

缓存数据存储在本地数据库的 `response_cache` 表中：

| 字段 | 说明 |
|------|------|
| `cache_key` | 统一生成的缓存标识（由 prompt + format + chartType + model + configName + contextHash 组合生成） |
| `prompt` | 用户输入的描述文本 |
| `format` | 图表格式 |
| `chart_type` | 图表类型 |
| `model` | AI 模型标识 |
| `config_name` | LLM 配置名称 |
| `context_hash` | 对话上下文哈希 |
| `response` | 生成的图表代码 |
| `created_at` | 创建时间 |
| `last_used_at` | 最后使用时间 |
| `use_count` | 使用次数 |
| `ttl` | 生存时间（秒） |

### L1 缓存

L1 内存缓存不持久化，使用标准的 Map + LRU 策略实现，进程重启后自动清空。
